# Lean-TCB: Trust Boundary Classifier — Design Document

*Created: 2026-02-21*

## Problem

When building verified software in Lean 4, every proof is kernel-checked — but that doesn't mean every declaration is equally trustworthy. The kernel verifies that proofs match their stated types, but it cannot verify that *definitions* capture the intended semantics. If `prime` is accidentally defined as `n > 3`, the kernel will happily prove theorems about this wrong definition.

The trust question is: **"which declarations must a human review to believe the top-level theorems are meaningful?"**

No tool in any ITP ecosystem answers this question systematically.

## Core Semantic Model

The kernel checks everything, but trust relevance depends on declaration kind:

| Kind | Type (signature) | Value (body/proof) | Rationale |
|------|-----------------|--------------------|----|
| `axiom` | Spec | *(none)* | Pure assumption, no proof to check |
| `def` | Spec | **Spec** | Body defines what the function *means* |
| `theorem` | Spec | Proof infra | Proof is kernel-checked; proof irrelevance |
| `opaque` | Spec | Proof infra | Body is hidden from downstream consumers |
| `inductive` | Spec | N/A | Constructor types define the type's meaning |

**Key insight:** For `def`s, the body IS the specification. For `theorem`s, only the statement matters — the proof term is verified by the kernel and need not be trusted.

### Edge Cases

- **`abbrev`**: Lean represents these as `defnInfo` with `ReducibilityHints.abbrev`. We walk type + body (same as `def`).
- **`instance`**: Also `defnInfo` with an attribute. Walk type + body.
- **`class`/`structure`**: Represented as `inductInfo`. Walk constructor types.
- **`noncomputable def`**: Still `defnInfo`. Body still defines meaning.
- **Mutual definitions**: The `all` field lists co-defined names. We enqueue the entire block.
- **Constructors**: `ctorInfo` — we also enqueue the parent inductive.
- **Recursors**: `recInfo` — walk type only, enqueue parent inductive(s).

## Algorithm

```
Input:  entry-point theorem names E₁, ..., Eₙ
Output: partition of reachable declarations into spec / proof-infrastructure

1. queue = [E₁, ..., Eₙ], visited = ∅, spec = ∅
2. While queue not empty:
   a. Pop name N; skip if visited; mark visited
   b. Look up ConstantInfo for N
   c. Walk N.type → collect all referenced constants → enqueue
   d. Per kind:
      - defnInfo:   also walk N.value (body defines meaning)
      - thmDecl:    do NOT walk value (proof is kernel-checked)
      - axiomDecl:  nothing more (no value)
      - opaqueDecl: do NOT walk value (hidden)
      - inductDecl: walk constructor types
      - ctorInfo:   enqueue parent inductive
      - recInfo:    walk type only, enqueue parent inductives
   e. For mutual blocks: enqueue all members
   f. Add N to spec set
3. proof_infra = all environment declarations NOT in spec
```

We use `Expr.foldConsts` to extract constant references from expressions.

## Known Limitations

### Over-approximation of Def Bodies

When walking a `def` body, `foldConsts` collects ALL referenced constants, including proof sub-terms (e.g., decidability witnesses in `if h : ...`). This over-approximates the spec set — some proof-infrastructure constants get classified as specification.

This is the **safe direction** (never misses a real dependency). A v2 refinement could do type-directed traversal, skipping sub-expressions whose type is `Prop`.

### Sorry Detection

Since we intentionally skip theorem proof terms, we will NOT discover `sorryAx` through our traversal. The tool answers "what must a human trust?" — not "is the proof complete?" These are orthogonal concerns. A separate sorry check can be added as a warning.

### Library Noise

The spec set includes all transitively-reachable library declarations (e.g., `Nat.add`, `HAdd.hAdd`). While technically part of the TCB, these are well-established core definitions. The output groups them separately from user-defined specification to keep the report actionable.

## `@[tcb]` Annotation Checking

*Added: 2026-02-21*

Users can mark declarations with `@[tcb]` to declare their expected TCB:

```lean
@[tcb] def myPred (n : Nat) : Prop := n > 0
@[tcb] theorem main_thm : myPred 1 := ...
```

When `#tcb` runs, it cross-checks the computed spec set against these annotations:

- **Unannotated TCB members**: a declaration is in the computed TCB but not marked `@[tcb]`. This means an unexpected trust dependency crept in.
- **Unnecessary annotations**: a declaration is marked `@[tcb]` but not in the computed TCB for the given entry points. This may indicate a stale annotation or a different entry point is needed.

Parent coverage: annotating an `inductive` covers its constructors, recursor, and auto-generated companions (`.casesOn`, `.brecOn`, `.match_1`, etc.). This prevents noisy warnings for declarations the user cannot directly annotate.

The check only activates when at least one `@[tcb]` annotation exists in the current module. Without annotations, the output is unchanged.

## Module Structure

- **`Types.lean`** — `TcbResult`
- **`Classify.lean`** — `trustRelevantExprs` (per-kind expression selection), `collectConstants`, mutual/ctor/rec helpers
- **`Core.lean`** — `computeTcb` worklist traversal
- **`Attr.lean`** — `@[tcb]` tag attribute registration, `isTcbAnnotated`, `checkAnnotations`
- **`Format.lean`** — Groups results into axioms/user-spec/library-spec/summary; uses `getModuleIdxFor?` to distinguish user vs imported declarations
- **`Command.lean`** — `#tcb` / `#tcb!` elaboration commands with name resolution and annotation checking
